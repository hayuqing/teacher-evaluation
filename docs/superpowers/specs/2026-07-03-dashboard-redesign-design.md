# 数据看板重构设计

## 目标

重构管理端「数据看板」页面，使其从静态概览页升级为面向校长、教务主任和管理者的教师评价分析页。页面需要基于学生端提交的评价数据，对教师表现进行多维分析，并提供按年级查看教师整体情况的能力。

当前阶段仍是静态 HTML 原型。实现时优先读取浏览器本地存储中的学生提交数据；如果没有真实提交数据，则自动使用内置模拟数据。后续前后端分离时，只替换数据获取函数即可。

## 用户与使用场景

主要用户是学校管理者。他们进入看板时最关心三类问题：

1. 评价收集是否充分：哪些年级、班级或教师的评价样本不足。
2. 教师整体表现如何：全校、年级、学科维度的平均表现和差异。
3. 哪些教师或维度需要关注：低分教师、短板维度、异常波动或样本不足。

看板不作为单个学生反馈明细页，不展示学生身份，也不暴露可追溯个人的信息。

## 页面信息架构

### 顶部筛选区

筛选区位于页面标题下方，提供以下控件：

- 评价任务：默认「2026 春季学生评教」
- 学段：全部、小学低年级、小学高年级、初中
- 年级：全部、一至九年级
- 学科：全部及任课安排中的动态学科
- 数据来源提示：真实提交数据或演示模拟数据

筛选会驱动整页指标、图表和明细表刷新。当前原型中筛选在前端本地完成。

### 核心指标区

指标卡展示管理者首先需要看的关键结果：

- 参与教师：当前筛选范围内有评价样本的教师数量
- 有效评价：学生对教师的有效评价条数
- 平均得分：按百分制展示，保留一位小数
- 完成率：有效评价条数 / 理论应评价条数
- 需关注教师：低于关注阈值或样本不足的教师数量

关注阈值在原型中固定为 75 分以下，样本不足阈值固定为少于 8 条评价。

### 年级整体分析

这是页面主分析区。按年级展示：

- 年级平均分
- 参与学生数
- 参与教师数
- 完成率
- 优势维度
- 短板维度
- 状态标签：优秀、稳定、需关注、样本不足

该区包含一个年级对比柱状图和一组年级分析卡。柱状图用于快速比较不同年级教师整体得分；分析卡用于解释每个年级的主要结论。

### 多维图表区

图表区提供四类视角：

- 年级平均分对比：横向柱状图
- 评价维度表现：按教学清晰度、课堂管理、启发思维、关爱公平、作业反馈、责任态度等维度展示
- 学科平均分对比：显示不同学科的平均得分和样本量
- 教师得分分布：将教师分为 90 分以上、80-89、70-79、70 分以下

图表使用原生 HTML/CSS/SVG 实现，不引入外部图表库。这样静态原型轻量，也便于后续迁移。

### 教师分析明细

底部表格展示教师粒度数据：

- 教师姓名
- 学科
- 任教年级/班级
- 评价人数
- 平均得分
- 最高维度
- 短板维度
- 状态标签

支持按顶部筛选和关键词搜索联动。状态标签规则：

- 优秀：平均分 >= 90 且样本充足
- 稳定：平均分 >= 75 且样本充足
- 需关注：平均分 < 75 且样本充足
- 样本不足：评价样本 < 8

## 数据模型

### 学生端提交数据

学生端最终提交时，将每位教师的评价结果写入浏览器本地存储。建议键名为：

```text
teacherEvaluationSubmissions
```

每条提交记录结构：

```json
{
  "id": "submission-20260703-001",
  "submittedAt": "2026-07-03T10:32:00+08:00",
  "tier": "high",
  "grade": "8",
  "gradeName": "八年级",
  "className": "八年级（1）班",
  "teacherName": "方民主",
  "subject": "英语",
  "answers": {
    "0": 0,
    "1": 1,
    "2": 0,
    "3": 0,
    "4": 1,
    "5": 0
  },
  "comment": "讲解很清楚"
}
```

`answers` 中的值沿用学生端选项索引。统计时按学段量表选项数量映射为百分制分数。

### 模拟数据

如果本地存储没有可用提交数据，管理端生成演示数据。模拟数据基于现有任课安排生成，覆盖小学和初中，保留教师、学科、年级、班级之间的关系。模拟数据用于演示页面能力，不作为真实结果。

### 计算口径

单个维度得分：

- 选项越靠前分数越高
- 3 档量表：100、75、50
- 4 档量表：100、85、70、55
- 5 档量表：100、87.5、75、62.5、50

单条教师评价得分是各维度平均分。教师平均分是该教师全部评价得分的平均值。年级、学科、全校平均分均按评价条数加权平均。

完成率在当前静态原型中按以下方式估算：

```text
有效评价条数 / 理论评价条数
```

理论评价条数来自任课安排中的班级人数乘以该班需要评价的教师数。没有完整映射时，使用模拟数据中的目标量作为兜底。

## 数据流

1. 页面进入数据看板时调用 `renderDashboard()`。
2. `getDashboardSubmissions()` 优先读取 `localStorage.teacherEvaluationSubmissions`。
3. 如果没有真实数据，调用 `buildMockDashboardSubmissions()` 生成模拟数据。
4. `buildDashboardModel(filters)` 统一计算指标、年级分析、维度表现、学科表现、教师明细。
5. 各渲染函数只消费计算后的模型，不直接读取原始数据。

后续接后端时，只需要将 `getDashboardSubmissions()` 改为异步 API 数据源，并保持输出结构一致。

## 交互行为

- 切换学段后，年级下拉只显示对应年级。
- 切换任一筛选项后，所有指标、图表和表格同步刷新。
- 搜索教师时，匹配教师姓名、学科、年级、班级。
- 点击年级分析卡会将年级筛选切到该年级。
- 数据来源提示显示「真实提交数据」或「演示模拟数据」。
- 若真实数据为空，页面仍展示完整模拟看板，避免空白原型。

## 视觉方向

页面应保持现有管理端风格：安静、清晰、适合重复查看。避免营销式大 hero 和装饰性图形。信息密度应高于当前首页，但保持分区清楚。

布局建议：

- 顶部：标题、导出按钮、创建任务按钮
- 第二行：筛选工具条
- 第三行：5 个紧凑指标卡
- 主体：左侧年级整体分析，右侧维度/学科分析
- 底部：教师明细表和异常提醒

图表颜色应避免单一色系，以绿色、蓝色、橙色、红色搭配表达不同状态。

## 错误与空状态

- 本地存储 JSON 解析失败时，忽略损坏数据并回退到模拟数据。
- 筛选后无数据时，显示空状态文案「当前筛选范围暂无评价数据」。
- 样本不足的教师和年级不直接判定为低分，优先标记为「样本不足」。

## 验证方式

实现完成后需要验证：

- `admin-system.html` 中脚本可通过 `node --check`
- 无真实提交数据时，数据看板能显示模拟数据
- 存在本地存储提交数据时，看板优先使用真实数据
- 学段、年级、学科、搜索筛选能刷新指标和表格
- 点击年级分析卡能切换年级筛选
- Git diff 中数据看板重构与学生端提交存储改动范围清晰

## 非目标

当前阶段不做：

- 后端 API
- 用户登录鉴权
- 真实数据库存储
- Excel/PDF 正式报表导出
- 复杂图表库接入
- 学生个人身份追踪
